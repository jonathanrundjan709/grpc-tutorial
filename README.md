# Rust gRPC Tutorial Reflection

1. Unary RPC handles one request and one response, so it fits actions like payment submission or login. Server streaming still starts from one client request, but the server returns many responses over time, which is suitable for transaction history, logs, or notifications. Bi-directional streaming allows both sides to keep sending messages independently, making it the best fit for chat, live collaboration, and interactive real-time systems.

2. A Rust gRPC service must protect identity, permissions, and transport security. Authentication can be added through tokens, API keys, or mTLS; authorization should verify whether a caller may access a resource; and data encryption should use TLS so traffic is not exposed in transit. Input validation, rate limiting, and secure secret storage are also important because binary protocols are still vulnerable if the surrounding system is weak.

3. Bidirectional streaming is harder because the client and server stay connected for longer periods and must coordinate concurrent reads and writes correctly. Common issues include backpressure, dropped connections, message ordering, partial failures, resource leaks, and difficulty shutting down streams cleanly. In chat-style applications, developers also need to think about reconnection strategy, presence state, and how to recover when one side disconnects unexpectedly.

4. `tokio_stream::wrappers::ReceiverStream` is useful because it turns a Tokio channel into a stream that Tonic can send to the client with minimal boilerplate. This makes the code easy to read, easy to compose with async tasks, and practical for streaming generated events. Its downside is that the design depends on channel buffering and task coordination, so poor sizing or slow consumers can increase memory use, latency, or dropped work if not handled carefully.

5. The code can be made more reusable by separating protobuf definitions, transport handlers, business logic, and shared models into different modules. For example, each service implementation can live in its own file, while shared validation, error handling, and configuration live in common modules. This structure reduces duplication, makes testing easier, and allows future services to reuse the same patterns without rewriting core logic.

6. More complex payment processing would need validation, idempotency checks, persistence, and integration with external systems. A real implementation should verify account balance, confirm currency and amount rules, talk to a payment gateway, store transaction records, and handle rollback or retry behavior on failure. It should also log audit trails and return structured business errors instead of only sending a simple success message.

7. Using gRPC pushes distributed systems toward strongly typed contracts and more explicit service boundaries. That improves consistency, code generation, and performance between internal services, but it can reduce interoperability with environments that expect simple JSON over HTTP. Teams often need API gateways or adapter layers when gRPC services must communicate with browsers, third-party platforms, or legacy systems.

8. HTTP/2 gives gRPC multiplexing, header compression, persistent connections, and efficient streaming, so it usually performs better than HTTP/1.1 for service-to-service communication. However, HTTP/2 debugging can be less straightforward, and gRPC is not as universally browser-friendly as plain REST over HTTP/1.1. Compared with WebSocket, gRPC provides stronger schemas and RPC semantics, while WebSocket can be simpler when an application only needs raw two-way messaging.

9. REST mainly follows a request-response model where the client asks for data and waits for each separate reply, which is simple but less efficient for highly interactive updates. gRPC bidirectional streaming keeps a connection open so both sides can exchange messages continuously with lower latency. That makes gRPC more responsive for real-time features such as chat, telemetry, and collaborative applications.

10. Protocol Buffers enforce a clear schema, which improves type safety, compactness, validation, and compatibility management across services. JSON is more flexible and easier to inspect manually, but that flexibility can lead to inconsistent payload structures and weaker contracts if teams are not disciplined. In practice, Protobuf is better for performance-sensitive internal systems, while JSON is often easier for public-facing APIs and ad hoc integrations.
