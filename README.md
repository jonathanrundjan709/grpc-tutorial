# Rust gRPC Tutorial Reflection

1. Unary RPC menangani satu request dan satu response, sehingga cocok untuk aksi seperti pengiriman pembayaran atau login. Server streaming tetap dimulai dari satu request client, tetapi server dapat mengirim banyak response secara bertahap. Pola ini cocok untuk riwayat transaksi, log, atau notifikasi. Sementara itu, bi-directional streaming memungkinkan client dan server saling mengirim pesan secara independen, sehingga paling cocok untuk chat, kolaborasi langsung, dan sistem real-time yang interaktif.
2. Layanan gRPC di Rust harus melindungi identitas, hak akses, dan keamanan transportasi data. Authentication dapat ditambahkan melalui token, API key, atau mTLS. Authorization perlu memastikan apakah pemanggil layanan memang berhak mengakses suatu resource. Selain itu, enkripsi data perlu menggunakan TLS agar traffic tidak terekspos saat dikirim melalui jaringan. Validasi input, rate limiting, dan penyimpanan secret yang aman juga penting karena protokol binary tetap bisa rentan jika sistem di sekitarnya lemah.
3. Bidirectional streaming lebih sulit karena client dan server mempertahankan koneksi dalam waktu yang lebih lama, serta harus mengatur proses membaca dan menulis pesan secara concurrent dengan benar. Masalah umum yang bisa muncul meliputi backpressure, koneksi terputus, urutan pesan, partial failure, resource leak, dan kesulitan menutup stream dengan bersih. Dalam aplikasi seperti chat, developer juga perlu memikirkan strategi reconnect, status presence, dan pemulihan ketika salah satu sisi terputus secara tiba-tiba.
4. tokio_stream::wrappers::ReceiverStream berguna karena dapat mengubah Tokio channel menjadi stream yang bisa dikirim oleh Tonic ke client dengan boilerplate yang minimal. Hal ini membuat kode lebih mudah dibaca, mudah dikombinasikan dengan async task, dan praktis untuk melakukan streaming event yang dihasilkan secara dinamis. Kekurangannya adalah desain ini bergantung pada buffering channel dan koordinasi task, sehingga ukuran buffer yang buruk atau consumer yang lambat dapat meningkatkan penggunaan memori, latency, atau menyebabkan pekerjaan terbuang jika tidak ditangani dengan baik.
5. Kode dapat dibuat lebih reusable dengan memisahkan definisi protobuf, handler transport, business logic, dan shared model ke dalam modul yang berbeda. Misalnya, setiap implementasi service dapat ditempatkan di file masing-masing, sementara validasi bersama, error handling, dan konfigurasi dapat ditempatkan di modul umum. Struktur seperti ini mengurangi duplikasi, mempermudah testing, dan memungkinkan service baru menggunakan pola yang sama tanpa menulis ulang logic inti.
6. Pemrosesan pembayaran yang lebih kompleks membutuhkan validasi, pengecekan idempotency, persistence, dan integrasi dengan sistem eksternal. Implementasi nyata harus memverifikasi saldo akun, memastikan aturan mata uang dan jumlah pembayaran, berkomunikasi dengan payment gateway, menyimpan catatan transaksi, serta menangani rollback atau retry ketika terjadi kegagalan. Sistem juga perlu mencatat audit trail dan mengembalikan business error yang terstruktur, bukan hanya response sederhana berupa status sukses.
7. Penggunaan gRPC mendorong distributed system ke arah kontrak yang strongly typed dan batas antarservice yang lebih eksplisit. Hal ini meningkatkan konsistensi, code generation, dan performa komunikasi antarservice internal. Namun, gRPC dapat mengurangi interoperabilitas dengan environment yang lebih terbiasa menggunakan JSON sederhana melalui HTTP. Tim sering kali membutuhkan API gateway atau adapter layer ketika service gRPC harus berkomunikasi dengan browser, platform pihak ketiga, atau sistem legacy.
8. HTTP/2 memberikan gRPC kemampuan multiplexing, header compression, persistent connection, dan streaming yang efisien, sehingga biasanya lebih cepat untuk komunikasi service-to-service dibandingkan HTTP/1.1. Namun, debugging HTTP/2 bisa lebih sulit, dan gRPC tidak se-universal REST biasa dalam konteks browser. Dibandingkan WebSocket, gRPC menyediakan schema yang lebih kuat dan semantik RPC yang lebih jelas, sedangkan WebSocket bisa lebih sederhana jika aplikasi hanya membutuhkan komunikasi dua arah mentah.
9. REST umumnya mengikuti model request-response, yaitu client meminta data dan menunggu response terpisah untuk setiap permintaan. Model ini sederhana, tetapi kurang efisien untuk update yang sangat interaktif. gRPC bidirectional streaming mempertahankan koneksi terbuka sehingga kedua sisi dapat bertukar pesan secara terus-menerus dengan latency yang lebih rendah. Hal ini membuat gRPC lebih responsif untuk fitur real-time seperti chat, telemetry, dan aplikasi kolaboratif.
10. Protocol Buffers menerapkan schema yang jelas, sehingga meningkatkan type safety, ukuran payload yang ringkas, validasi, dan manajemen kompatibilitas antarservice. JSON lebih fleksibel dan lebih mudah dibaca secara manual, tetapi fleksibilitas tersebut dapat menyebabkan struktur payload yang tidak konsisten dan kontrak yang lebih lemah jika tim tidak disiplin. Dalam praktiknya, Protobuf lebih cocok untuk sistem internal yang sensitif terhadap performa, sedangkan JSON sering lebih mudah digunakan untuk API publik dan integrasi ad hoc.
